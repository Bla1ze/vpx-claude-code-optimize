# darkchaos.vbs

## CRITICAL: Do Not Hallucinate Names

This project is a single 75K+ line VBS file where every identifier ŌĆö event names, sound names, show names, light names, shot names, timer names ŌĆö is a **plain string**. There is no compiler, no autocomplete, no type checker. If you invent a name that doesn't exist, VPX silently ignores it or crashes at runtime. This is the #1 source of bugs when editing this file.

**Why this codebase causes hallucinations:**
- String-based DSL looks like it follows obvious naming patterns, but it doesn't always. `flicker_color_on_intensity` exists but `flicker_color_on` is a different show with different behavior. `flicker2_color_on_intensity` is yet another variant. Easy to guess wrong.
- Sound names like `voc_critically_low_health`, `sfx_powerdown`, `sfx_jackpot` look like they could be anything. You cannot invent sound names ŌĆö they must match actual audio assets. There is no manifest; the only source of truth is what's already used in the file.
- Light group names (`GI`, `LStars`, `LShip`, `tInlaneGI`) and individual GI lamps (`gi01`..`gi22`, `giapron`) are specific to this table. Not all numbers exist (e.g. there may not be a `gi02`). Check existing usage.
- Event names are generated by convention (`mode_<name>_started`, `timer_<name>_complete`, etc.) but custom events are freeform strings. If you type `play_sfx_explosion` but the SoundPlayer entry says `play_sfx_boom`, nothing plays and there's no error.
- The file has ~40+ mode subs with similar but subtly different patterns. Easy to copy from the wrong template and carry over names that belong to a different mode.

**Rules to follow:**
1. **ALWAYS grep/search for a name before using it.** Never assume a sound, show, light, or event exists. Verify it appears in the file.
2. **When adding new sounds to a SoundPlayer block, only reference `.Sound` values that already exist elsewhere in the file.** Search for the exact string.
3. **When referencing shows in ShowPlayer, search `CreateGlfShow("name")` to confirm the show exists.** The full list is at line ~33000+.
4. **When referencing light names, check existing ShowPlayer/LightPlayer usage** to confirm the light group or individual lamp name is real.
5. **Copy event name strings exactly from the EventPlayer where they're defined** rather than retyping them from memory. One wrong character = silent failure.
6. **If you're unsure whether something exists, read the relevant section of the file first.** A 30-second search saves a runtime crash.

## GLF Framework Rules

**ShowPlayer: every EventName must be unique.** `ShowPlayer.EventName(name)` adds to an internal Dictionary keyed by the raw event string. If you reuse the same event name for multiple ShowPlayer entries, VPX throws "Class not registered" at runtime (line ~60329). When you need one event to start multiple shows (e.g. 8 GI lamps), give each entry its own event name (`init_bo_surge_gi01`, `init_bo_surge_gi05`, etc.) and fire them all from a single EventPlayer `.Add` array.

**Shows are declarative fixed-step loops.** Each show created with `CreateGlfShow` is a sequence of `.AddStep(Null, Null, duration)` steps where duration is in seconds (e.g. `.16` = 160ms). Steps set `(lights)|intensity|(color)` per step. Duration `-1` means "hold forever" (used for final resting states). The `.Speed` property on the ShowPlayer entry multiplies playback rate. Shows loop infinitely by default; set `.Loops = 1` for one-shot, `.Loops = -1` explicitly for infinite.

**Faking randomness with shows.** Since shows are periodic loops, apparent randomness comes from layering multiple shows on different keys at coprime speeds (3, 4, 5, 6, 7). With different step counts and speeds, the combined visual won't visibly repeat. Use a long custom show (12-16 irregular steps) as the base layer.

**Show definitions live at line ~33000+.** Built-in shows: `led_color` (static color), `flash_color` (on/off strobe), `flicker_color` (6-step intensity wobble), `flicker2_color_on_intensity` (12-step wobble, ends at token intensity), `flicker_color_off` (flicker then fade off), `off` (lights off). Custom shows go in this same region.

**EventPlayer vs ShowPlayer vs RandomEventPlayer.** EventPlayer maps events to arrays of other events. ShowPlayer maps events to show configurations (one entry per event name). RandomEventPlayer maps one event to weighted random choices of other events. All three use unique event name keys ŌĆö no duplicates within a single player block.

**Shot profiles.** The shared `powerups` profile (line ~5610) has states: 0=unlit, 1=ready, 2=collected. Hit conditions use `current_player.shot_<modename><N> == 1` to only collect when ready. Set state 1 on mode start, state 2 on hit. States have `.Priority = 50` to override competing modes on the same lights (see Light Stack Priority below).

## Light Stack Priority

**How priority works.** Each light has a priority stack (`GlfLightStack`). The highest-priority active entry wins. Effective priority = `mode_priority + show_entry_priority`. If no `.Priority` is set on a ShowPlayer entry or shot profile state, it defaults to 0, so effective priority equals the mode's priority.

**Always set `.Priority` on shot profile states** that need to override other modes' shows on the same lights. Without it, a lower-priority mode with `.Priority > 0` can have a higher effective priority than a higher-priority mode with `.Priority = 0`.

**Concrete example of the trap:** alien_attack (mode 500) has `.Priority = 10` on its `lit` state ŌåÆ effective 510. A post-wave mode at priority 507 with no `.Priority` ŌåÆ effective 507. The lower-priority mode wins because 510 > 507.

**Modes that compete for main shot lights (LS1-LS6, LPC4, LPC5) during gameplay:**

| Mode | Mode Priority | Show Priority | Effective | When active |
|------|--------------|---------------|-----------|-------------|
| base | 200 | 0 | 200 | Always (ball start ŌåÆ ball end) |
| alien_attack | 500 | 10 (lit) | 510 | new_ball_active / stop_meteor_wave; does NOT stop for post-wave modes |
| solar_flare | 505 | 50 (powerups) | 555 | start_solar_flare ŌåÆ stop/timeout |
| blackout | 506 | 50 (powerups) | 556 | start_blackout ŌåÆ stop/timeout |
| incursion | 507 | 50 (powerups) | 557 | start_incursion ŌåÆ stop/timeout |
| training_qualify | 513 | 0 | 513 | new_ball_started / stop_meteor_wave; does NOT stop for post-wave modes |

**The `.Priority = 50` on `powerups` ensures post-wave modes (555-557) override alien_attack (510), training_qualify (513), and combos (530).**

## Shot State Variables and RestartEvents

**Player variables persist across mode runs.** Shot state is stored in `current_player.shot_<shotname>` (e.g., `current_player.shot_solar_flare1`). These are player-scoped variables that survive mode stop/start cycles. If a mode runs twice in one ball, shots start with stale state from the previous run.

**Always add `.RestartEvents` to shots in repeatable modes.** This resets the shot to state 0 when the event fires. Without it, conditional checks like `shot_incursionN == 0` fail on the second run because the variable is still 1 or 2.

```
With .Shots("my_shot"&x)
    .Profile = "powerups"
    ' ... tokens, control events ...
    .RestartEvents = Array("mode_my_mode_started")
End With
```

**For modes with tracking variables** (like incursion's `incursion_activated`, `incursion_collected`, `incursion_path`), also reset them via VariablePlayer on `mode_<name>_started`:

```
With .VariablePlayer()
    With .EventName("mode_incursion_started")
        With .Variable("incursion_path") : .Action = "set" : .Int = 0 : End With
        With .Variable("incursion_activated") : .Action = "set" : .Int = 0 : End With
        With .Variable("incursion_collected") : .Action = "set" : .Int = 0 : End With
    End With
End With
```

**Reference: alien_attack uses `.RestartEvents`** on its shots (line ~5846) ŌĆö follow this pattern for any mode with shots that can run more than once per ball.

## Conditional Events

**Syntax:** `"event_name{condition}"` ŌĆö the condition is evaluated when the event fires. If false, the entry is skipped silently. Used in EventPlayer `.Add` keys and shot `.ControlEvents().Events`.

**Common patterns:**
- `MainShotNames(x-1)&"_hit{current_player.shot_mymode"&x&" == 1}"` ŌĆö only process hit if shot is in ready state
- `"timer_mymode_tick{devices.timers.mymode.ticks == 24}"` ŌĆö fire on a specific tick count
- `"timer_mymode_tick{current_player.some_var == 1}"` ŌĆö per-tick conditional check
- `"incursion_activate_shot1{current_player.shot_incursion1 == 0}"` ŌĆö only activate if currently unlit

**Multiple conditions** use `&&`: `"event{cond1 && cond2 && cond3}"`

**Each unique condition string is a separate EventPlayer key**, so you can have multiple entries for the same base event with different conditions (e.g., 8 entries for `timer_incursion_tick` with different `shot_incursionN == 1` checks).

## Mode Construction Pattern

```
Sub CreateMyMode
    With CreateGlfMode("mode_name", priority)
        .StartEvents = Array("start_mode_name")
        .StopEvents = Array("stop_mode_name","start_meteor_wave",...)
        With .EventPlayer()  ...  End With
        With .RandomEventPlayer()  ...  End With   ' optional
        With .SoundPlayer()  ...  End With
        With .Shots("name") / .ShotGroups("name")  ...  End With
        With .Timers("name")  ...  End With
        With .SegmentDisplayPlayer()  ...  End With
        With .ShowPlayer()  ...  End With
    End With
End Sub
```

Modes auto-generate events: `mode_<name>_started`, `mode_<name>_stopping`. Timers generate `timer_<name>_tick` and `timer_<name>_complete`.

## VBScript Performance Optimization Rules

This table runs a 10ms game loop. Every per-frame operation is multiplied by ball count, light count, or source count. Seemingly trivial overhead becomes a bottleneck at scale. These rules come from real bugs and performance issues found in this codebase.

### VBScript Language Traps

**Never self-reference a Variant array element during assignment.** VBScript may clear the destination before evaluating the source. This causes runtime errors or silent data loss:
```vbs
' BROKEN ŌĆö m_compiled(f) may be cleared before reading m_compiled(f)(0)
m_compiled(f) = Array(m_compiled(f)(0), m_compiled(f)(1), m_compiled(f)(2), n, newData)

' CORRECT ŌĆö cache into locals first, then assign
Dim old: old = m_compiled(f)
m_compiled(f) = Array(old(0), old(1), old(2), n, newData)
```
This applies to any `arr(i) = Array(arr(i)(0), ...)` pattern. Always cache into a local variable first.

**VBScript arrays are value types inside Variants.** When you do `x = someArray(i)`, you get a *copy*. Modifying `x` does not modify the original. You must write the modified copy back: `someArray(i) = x`. This is different from objects (which use `Set` and are reference types). Always verify whether you're working with a copy or a reference when modifying nested array data.

**`Scripting.Dictionary` is expensive in hot loops.** `.Exists(key)` + `.Item(key)` is two hash lookups per call. In a loop over 90+ lights at 10ms intervals, Dictionary operations dominate CPU time. Prefer:
- Pre-resolved integer indices stored in parallel arrays (indexed by position, not name)
- Cache `.Keys()` into a local variable once per tick, not per iteration
- Use `.Item(key) = value` for in-place updates instead of `.Remove` + `.Add`

**`(new ClassName)` allocates on every call.** In per-tick loops, avoid creating new objects when you can reuse or pre-allocate them. Pre-built caches (like `BuildOffStateCache`) that create objects once at init and reuse them per-tick eliminate thousands of allocations/sec.

### Identifying Hot Paths

**Know what runs per-tick vs per-event.** In this codebase:
- `LStateController.Update()` runs every LampTimer tick (~10ms) ŌĆö this is the hottest path
- `DynamicBSUpdate` runs every frame ŌĆö hot for ball shadow math
- `RollingUpdate` runs every frame ŌĆö hot for ball rolling sounds
- `GameTimer_Timer` runs every 10ms ŌĆö flasher visibility, drop target animation
- Mode event handlers run on switch hits ŌĆö occasional, not hot

**Profile by multiplication, not by complexity.** A simple Dictionary lookup is O(1) but if called 90├Ś per tick at 100 ticks/sec, that's 9000 lookups/sec. A complex computation called once per ball start is irrelevant. Always ask: "how many times per second does this line execute?"

### The Lazy vs Eager Initialization Problem

**Identify when caches are populated vs when they're first consumed.** This is the #1 cause of "first ball is slow" bugs. The pattern:
1. Cache is *designed* to be computed at init time
2. But the code that populates it runs inside a method that's only called during gameplay
3. First gameplay frame pays the full computation cost

**Audit every cache by tracing the call chain to its trigger:**
- Module-level code (`Dim x : x = value`) ŌåÆ runs at script load Ō£ō
- Code inside `Table1_Init` ŌåÆ runs before first ball Ō£ō
- Code inside `RegisterLights` ŌåÆ runs during Table1_Init Ō£ō
- Code inside mode start/event handlers ŌåÆ runs during gameplay Ō£Ś (lazy)
- Code inside property setters called from gameplay code ŌåÆ runs during gameplay Ō£Ś (lazy)

**When adding pre-computation, always verify the trigger.** Read the actual call chain. Don't assume "this runs at init" ŌĆö grep for who calls it and when.

### Common Optimization Patterns in This Codebase

**Pre-computed constants pattern** (see DynBSFactor2, BS_d10, etc.):
```vbs
' At module level ŌĆö computed once at script load
Dim MyFactor2: MyFactor2 = MyFactor * MyFactor
```
Use for any expression involving only Const values or other module-level Dims.

**Pre-resolved index pattern** (see ResolveLights):
When a Dictionary maps string names to objects with integer `.Idx` properties, and a hot loop looks up by string name, pre-resolve the names to integer indices at registration/definition time. Store the indices in parallel arrays. The hot loop then skips Dictionary entirely.

**String concatenation avoidance** (see BallRollStr):
Pre-build `"prefix_" & i` strings into an array at init. Eliminates string allocation in per-tick loops.

**COM property caching** (see DynamicBSUpdate locals):
Cache `gBOT(s).X`, `gBOT(s).Y`, `gBOT(s).Z` into local variables at the top of each iteration. Each `.X` access is a COM property read across the VBS/VPX boundary. Locals are pure VBScript stack access.

**Guard writes** (see flasher visibility):
Only write a COM property when the value actually changes: `If newVal <> obj.visible Then obj.visible = newVal`. COM writes are expensive even when redundant.

### What NOT to Optimize

- **One-shot initialization code** (Table1_Init, mode creation subs) ŌĆö runs once, not hot
- **Event handlers that fire on switch hits** ŌĆö runs a few times per second at most
- **String operations in mode definition** (EventPlayer setup, shot configuration) ŌĆö runs once at mode creation
- **Code that's already behind a conditional guard** (`If hasOverride = False Then`) ŌĆö only optimize the branch that actually executes in the common case
